Three ways to build the same app — I'll build each one up, box by box. Classic SSR: logic on the server, HTML over the wire. Simple, SEO-friendly, no build step — but every click is a full page reload: flicker, lost scroll, lost focus. SPA: gorgeous UX, but count the cost. Your logic, validation and routing now live on BOTH sides. You build and version a JSON API in the middle. You add a bundler and a build step. You keep client-side state in sync with the server. First paint and SEO get harder. And it's often two codebases, sometimes two languages — twice the places for a bug to hide. Alpine AJAX: the server still renders HTML over the wire — Alpine just swaps in the fragment that changed. You get the SPA feel with server logic, one codebase, one source of truth, and no build step. HTML over the wire, not JSON.

The pitch, in one image. Option one — a JSON API, client-side state, a build pipeline: the whole SPA stack, just to update part of a page. Option two — add x-target to a form. Logic stays on the server, the UX still feels instant. The server is the best place for your logic; send HTML, not JSON.

First, the old way. A plain form posts; the server responds and the browser throws the page away and renders a brand-new one. Note the request type — "document" — and the response is the entire HTML page. It works, but it flickers and you lose scroll position and focus.

Same form, one attribute: x-target="notes". On submit (click) Alpine fires a real AJAX request — POST, with the X-Alpine-Request header, and the response is text/html: just the #notes fragment, not JSON, not the whole page. (click) Alpine reads it, finds #notes, and (click) swaps that element in place. No reload, no lost scroll, no flicker.
