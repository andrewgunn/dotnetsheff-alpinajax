Three ways to build the same app, one click each. Classic SSR: logic lives on the server — but every click is a full page reload. SPA: gorgeous UX, but your logic, validation and routing now live on both sides, with a JSON API to build and maintain in between. Often two codebases. Alpine AJAX: the server still renders HTML, Alpine swaps in just the changed bit. Server logic, SPA feel, one codebase, one source of truth.

The pitch

Logic on the server.

SPA feel on the client.

This is the whole argument. The server is the best place for your logic — your data, auth and rules already live there. So send HTML, the browser's native format, not JSON you have to re-assemble on the client. And you keep the SPA feel with a handful of attributes, not a framework: no build step, no client state to sync, no second codebase.

So how does it work?

Your markup — add one attribute

<form action="/notes" method="post"
      x-target="notes">
  <textarea name="Body"></textarea>
</form>
→
request
←
#notes HTML
Network
POST /notes 200 fetch
Request Headers
X-Alpine-Request: true
Response text/html
<ul id="notes"> <li>Chase signature</li> <li>Pay deposit</li> </ul>
Concretely. You write a normal form and add one attribute, x-target="notes". On submit (click) Alpine fires a real AJAX request — here's the DevTools Network tab: a POST, with the X-Alpine-Request header, and the response is text/html, an actual #notes fragment, not JSON. (click) Then: Alpine intercepts the submit, reads that HTML, finds #notes in it, and swaps just that element in. The server already rendered it — Alpine just plucks out the bit you named.
