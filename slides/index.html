SSR: logic on the server, full reloads. SPA: lovely UX but logic, validation and routing live TWICE, with a JSON API between. Alpine: server renders HTML, Alpine swaps the changed bits. One codebase, one source of truth.

So how does it work?

BEFORE — a plain Razor Pages form (full reload)

<form method="post" action="/notes">
  <textarea name="Body"></textarea>
</form>

AFTER — add one attribute

<form method="post" action="/notes" x-target="notes">
  <textarea name="Body"></textarea>
</form>

1 Alpine intercepts the submit   2 posts & reads the HTML response   3 finds #notes in it   4 swaps just that element in

No JSON. No client state. The server already renders the page; Alpine just plucks the element you named out of the response and swaps it. Same HTML, no full reload.

The pitch

The server is the best place for your logic.

Send HTML, not JSON.

Keep the SPA feel — with attributes, not a framework.

No build step. No client state to sync. No second codebase. ModelState validation, auth, domain rules all stay server-side. Alpine AJAX is ~5 attributes — that's the whole API.
